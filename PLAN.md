# PLAN.md — GitHub Actions Workflow Packer

Working name: **`actionspack`**

## 1. Vision

Build a **lockfile-first packer for GitHub Actions workflows**.

Developers should be able to author workflows conveniently with floating refs such as:

```yaml
- uses: actions/checkout@main
- uses: actions/setup-node@main
```

But CI should run a deterministic, fully packed workflow that does **not** resolve remote actions at runtime.

Core pitch:

> Pack your workflows. Lock your supply chain.

Or more explicitly:

> Write GitHub Actions with convenient refs like `@main`, but run workflows from a locked, fully inlined dependency graph.

## 2. Core principles

### 2.1 Lockfile-first

The lockfile is the source of truth.

The packed workflow is only a generated artifact.

```text
source workflows
      ↓
scan
      ↓
workflow.lock.yml
      ↓
pack
      ↓
packed workflows
```

The dependency graph, `tree`, `why`, `audit`, and `diff` commands should all read from the lockfile, not from the generated workflow bundle.

### 2.2 One lockfile for all workflow entries

There should be a single lockfile for the whole repository:

```text
.github/workflows/ci.src.yml
.github/workflows/release.src.yml
.github/workflows/deploy.src.yml
        ↓
.github/workflow.lock.yml
```

The lockfile records every entry workflow and its complete transitive dependency graph.

### 2.3 Default command is `pack`

The default command should be:

```bash
actionspack pack
```

Running the binary with no subcommand should behave the same way:

```bash
actionspack
```

Equivalent to:

```bash
actionspack pack
```

`install` should not be used as the main command name. The tool is not primarily installing packages; it is scanning, locking, inlining, and generating workflow artifacts.

### 2.4 Existing locked versions stay locked

Source workflows may use floating refs by default:

```yaml
- uses: actions/checkout@main
```

But once the lockfile resolves `main` to a commit SHA, `pack` should continue using that old locked SHA.

A dependency should only move to a newer commit when the user explicitly runs:

```bash
actionspack update
```

or:

```bash
actionspack update actions/checkout
```

This gives the package-manager behavior:

```text
source workflow says: @main
lockfile says: actual commit SHA
packed workflow uses: locked content from that SHA
```

### 2.5 Inline-only output mode

There should be no lock-only mode.

Pinning top-level actions to SHA is not enough, because a pinned action can still depend on other actions that use floating refs.

Therefore the packed workflow should satisfy this invariant:

```text
No remote GitHub Action resolution at runtime.
```

In practice, the generated workflow must not contain remote action references like:

```yaml
uses: owner/repo@ref
```

The tool should recursively resolve the full dependency graph, lock it, and inline or embed the resolved implementation into the packed workflow.

Unsupported cases should fail closed instead of silently emitting risky remote references.

## 3. Repository layout

Suggested default layout:

```text
.github/
  workflows/
    ci.src.yml
    release.src.yml
    ci.yml
    release.yml

workflow.lock.yml
```

Source workflows use a `.src.yml` suffix by default.

Generated workflows use the normal `.yml` suffix so GitHub Actions can execute them.

Example:

```text
.github/workflows/ci.src.yml   # authored by humans
.github/workflows/ci.yml       # generated by actionspack
workflow.lock.yml              # single lockfile for all entries
```

Optional config file:

```yaml
# actionspack.yml
entries:
  - source: .github/workflows/ci.src.yml
    output: .github/workflows/ci.yml
  - source: .github/workflows/release.src.yml
    output: .github/workflows/release.yml
```

If no config exists, the tool can discover:

```text
.github/workflows/*.src.yml
```

## 4. Command design

### 4.1 `pack`

Main command.

```bash
actionspack pack
```

Performs:

```text
1. Scan entry workflows.
2. Reuse existing lockfile resolutions when possible.
3. Resolve only new or missing dependencies.
4. Build or update the complete dependency graph in workflow.lock.yml.
5. Generate packed workflows from the lockfile.
6. Verify that generated workflows contain no remote action references.
```

Important behavior:

```text
pack does not update existing locked commits by default.
```

This means `@main` in the source workflow does not automatically move forward every time `pack` runs.

### 4.2 `scan`

```bash
actionspack scan
```

Scans source workflows and updates the lockfile graph without generating packed workflows.

`scan` should:

```text
- parse all entry workflows
- find direct action dependencies
- resolve missing dependencies
- parse action metadata
- recursively discover transitive dependencies
- write workflow.lock.yml
```

`scan` should not refresh already locked commits unless explicitly asked to do so.

### 4.3 `update`

```bash
actionspack update
```

Updates all dependencies according to their requested refs.

Examples:

```bash
actionspack update

actionspack update actions/checkout

actionspack update actions/setup-node --recursive
```

`update` should:

```text
1. Re-resolve selected requested refs, such as main, v4, or release branches.
2. Update resolved commit SHAs in workflow.lock.yml.
3. Re-scan transitive dependencies from the new resolved commits.
4. Optionally run pack afterwards.
```

Recommended default:

```text
actionspack update = update lockfile, then pack
```

A flag can disable packing:

```bash
actionspack update --lockfile-only
```

### 4.4 `tree`

```bash
actionspack tree
```

Shows dependency chains from the lockfile.

Example output:

```text
.github/workflows/ci.src.yml
└─ actions/setup-node@main
   ├─ actions/cache@v4
   └─ actions/github-script@v7
      └─ some/action@main
```

Useful options:

```bash
actionspack tree --entry .github/workflows/ci.src.yml

actionspack tree --json

actionspack tree --show-sha

actionspack tree --show-risk
```

### 4.5 `why`

```bash
actionspack why actions/cache
```

Shows why a dependency exists.

Example:

```text
actions/cache is used by:

.github/workflows/ci.src.yml
└─ actions/setup-node
   └─ actions/cache
```

This should be based entirely on `workflow.lock.yml`, not on the packed workflow.

### 4.6 `diff`

```bash
actionspack diff
```

Shows changes between the current lockfile and a previous lockfile state.

Example output:

```text
actions/setup-node
old: 49933ea5288caeca8642d1e84afbd3f7d6820020
new: 8f152de45cc393bb48ce5d89d36b731f54556e65

changed files:
- action.yml
- dist/setup/index.js

transitive changes:
- actions/cache: unchanged
- actions/github-script: updated
```

This command is important for reviewing supply-chain updates in pull requests.

### 4.7 `audit`

```bash
actionspack audit
```

Checks the lockfile and source workflows for security risks.

Potential checks:

```text
- remote actions not represented in the lockfile
- transitive actions using floating refs
- untrusted owners or repositories
- dependencies from forks
- Docker actions without digest pinning
- reusable workflows that cannot be safely inlined
- generated workflow contains remote uses
- lockfile content hash mismatch
```

### 4.8 `verify`

```bash
actionspack verify
```

Verifies that:

```text
- source workflows match the lockfile graph
- packed workflows match the lockfile
- packed workflows contain no remote action references
- no dependency is missing from the lockfile
```

This is the command to run in CI.

## 5. Lockfile design

The lockfile should be one file:

```text
workflow.lock.yml
```

It should record:

```text
- lockfile version
- all entry workflows
- output workflow paths
- direct dependencies for each entry
- every resolved package/action
- requested refs
- resolved commit SHAs
- action type
- transitive dependencies
- content digests
- risk metadata
```

Example:

```yaml
lockfileVersion: 1

entries:
  .github/workflows/ci.src.yml:
    output: .github/workflows/ci.yml
    dependencies:
      - package: github:actions/checkout//.
        requested: main
      - package: github:actions/setup-node//.
        requested: main

  .github/workflows/release.src.yml:
    output: .github/workflows/release.yml
    dependencies:
      - package: github:actions/checkout//.
        requested: main
      - package: github:softprops/action-gh-release//.
        requested: main

packages:
  github:actions/checkout//.:
    source: github
    owner: actions
    repo: checkout
    path: .
    requested: main
    resolved: 11bd71901bbe5b1630ceea73d27597364c9af683
    type: javascript
    contentDigest: sha256:example
    dependencies: []

  github:actions/setup-node//.:
    source: github
    owner: actions
    repo: setup-node
    path: .
    requested: main
    resolved: 49933ea5288caeca8642d1e84afbd3f7d6820020
    type: javascript
    contentDigest: sha256:example
    dependencies:
      - package: github:actions/cache//.
        requested: v4
        resolved: 0400d5f644dc74513175e3cd8d07132dd4860809
        foundAt: action.yml#runs.steps[0]

  github:actions/cache//.:
    source: github
    owner: actions
    repo: cache
    path: .
    requested: v4
    resolved: 0400d5f644dc74513175e3cd8d07132dd4860809
    type: javascript
    contentDigest: sha256:example
    dependencies: []
```

## 6. Dependency model

The dependency graph should include both direct and transitive dependencies.

Direct dependency:

```yaml
.github/workflows/ci.src.yml
└─ actions/setup-node@main
```

Transitive dependency:

```yaml
.github/workflows/ci.src.yml
└─ some/composite-action@main
└─ another/action@v1
└─ third/action@main
```

The tool should recursively inspect:

```text
- workflow step uses
- composite action steps
- nested composite action uses
- reusable workflow calls, if supported
- Docker image references, if supported
```

Every discovered dependency should be represented in the lockfile.

## 7. Inline-only packing behavior

The final generated workflow should not depend on GitHub resolving remote actions at runtime.

Forbidden in packed output:

```yaml
- uses: actions/checkout@main
- uses: actions/checkout@v4
- uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683
- uses: owner/repo/path@ref
```

Allowed forms depend on how strict the implementation is, but the preferred target is:

```text
- literal expanded workflow steps
- embedded scripts
- embedded action payloads restored at runtime from locked content
- no remote owner/repo@ref action references
```

For composite actions, inline by expanding steps directly into the workflow.

For JavaScript actions, possible strategy:

```text
1. Fetch the action content at the locked commit.
2. Compute and store content digest in the lockfile.
3. Embed the action payload into generated workflow steps.
4. Reconstruct the action payload in $RUNNER_TEMP during the job.
5. Execute the action's pre/main/post scripts as explicit steps.
```

For Docker actions:

```text
- Strict inline mode should fail unless the Docker dependency can be represented safely.
- A digest-pinned Docker image still requires a runtime pull, so it is not truly inline.
- Docker support should be treated as a policy decision, not silently allowed.
```

For reusable workflows:

```text
- Inline if the workflow can be safely transformed into local jobs/steps.
- Otherwise fail closed.
```

## 8. Security model

The tool should protect against supply-chain risk from remote action resolution.

Main risks addressed:

```text
- top-level floating refs like @main
- tag movement, such as @v4 being changed
- transitive action dependencies hidden inside composite actions
- pinned top-level actions that call unpinned transitive actions
- accidental workflow drift
```

Security invariants:

```text
1. All remote GitHub Actions dependencies are resolved to commit SHAs in the lockfile.
2. The full transitive graph is recorded.
3. Existing locked commits do not change during pack.
4. Updates are explicit.
5. Packed workflows contain no remote owner/repo@ref uses.
6. Unsupported dependency types fail closed.
7. tree, why, audit, and diff use the lockfile as the source of truth.
```

Important limitation:

```text
This tool can make action resolution deterministic, but it cannot prove that arbitrary action code is safe.
```

For example, a JavaScript action may still download files at runtime. The tool can audit and flag suspicious patterns, but it cannot fully eliminate runtime behavior risk without deeper sandboxing.

## 9. Example lifecycle

### 9.1 Author source workflow

```yaml
# .github/workflows/ci.src.yml
name: CI

on:
  push:

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@main
      - uses: actions/setup-node@main
        with:
          node-version: 22
      - run: pnpm install
      - run: pnpm test
```

### 9.2 Pack

```bash
actionspack pack
```

This generates or updates:

```text
workflow.lock.yml
.github/workflows/ci.yml
```

### 9.3 Inspect dependency tree

```bash
actionspack tree
```

Example:

```text
.github/workflows/ci.src.yml
├─ actions/checkout@main
│  resolved: 11bd71901bbe5b1630ceea73d27597364c9af683
└─ actions/setup-node@main
   resolved: 49933ea5288caeca8642d1e84afbd3f7d6820020
```

### 9.4 Explain a dependency

```bash
actionspack why actions/cache
```

Example:

```text
actions/cache is used by:

.github/workflows/ci.src.yml
└─ some/composite-action
   └─ actions/cache
```

### 9.5 Update dependencies

```bash
actionspack update
```

This explicitly moves locked commits forward.

Then review:

```bash
actionspack diff
```

Finally commit:

```text
workflow.lock.yml
.github/workflows/*.yml
```

## 10. MVP scope

Recommended MVP:

```text
- discover .github/workflows/src/*.yml
- parse workflow steps
- resolve GitHub action refs to commit SHAs
- generate one workflow.lock.yml
- recursively scan composite action dependencies
- implement tree
- implement why
- implement pack for composite actions
- verify packed output contains no remote uses
- fail closed for unsupported action types
```

MVP should prioritize correctness and safety over supporting every GitHub Actions feature.

## 12. Open design questions

### 12.1 What does inline mean for JavaScript actions?

Strictest interpretation:

```text
The generated workflow is a single file and contains everything needed to run the action.
```

Practical interpretation:

```text
The generated workflow embeds the locked action payload and reconstructs it during the job.
```

The project should decide whether local generated vendor directories are allowed. The current direction is stricter: avoid remote action resolution and prefer single-workflow packed artifacts.

### 12.2 Should Docker actions be supported?

Docker actions are difficult to make truly inline because the runner usually pulls images at runtime.

Possible policies:

```text
- strict: reject Docker actions
- digest-only: allow docker://...@sha256:...
- vendor-image: future advanced mode
```

### 12.3 Should `pack` automatically add new dependencies?

Recommended behavior:

```text
- existing locked dependencies stay locked
- new dependencies are resolved and added
- removed dependencies are pruned or marked unused
- updates require actionspack update
```

This matches package-manager ergonomics while keeping versions stable.

## 13. Summary

`actionspack` is a GitHub Actions dependency manager and packer.

It should:

```text
- let users author workflows with convenient refs like @main
- maintain one lockfile for all entry workflows
- record the full direct and transitive dependency graph
- support tree and why without relying on the generated bundle
- make pack the default command
- keep locked commits stable until explicit update
- only emit inline/packed workflows
- reject unsafe unresolved remote action dependencies
```

The key product idea:

> Pinning is not enough. The whole GitHub Actions dependency graph must be locked, visible, and packed.
